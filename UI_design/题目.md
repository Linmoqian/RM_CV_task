## Q1 - 自定义客户端界面开发

### 背景

你正在为 RoboMaster 机甲大师高校赛开发自定义操作员客户端。

比赛中，操作员通过客户端上的图传画面观看机器人摄像头回传的实时画面，同时需要随时掌握比赛状态和机器人的各项数据，并在必要时向机器人下达控制指令。图传画面占据了客户端窗口的主要区域。

请使用 Qt Quick 6 完成交互界面设计。

即使你之前没接触过 Qt Quick 也可以完成本题。Qt Quick 使用 QML 语言，它的语法接近 JSON/CSS，上手难度低，跟着文档即可快速入门。如果你用过 Qt Widgets，请注意 Quick 与之大相径庭：Widgets 侧重于“如何一步步构建界面”，而 Quick 侧重于描述“界面是什么样”，建议以全新视角学习。

参考资料：
- 官方文档：Qt Quick 入门、QML 文档、属性绑定、动画
- 推荐教程：QML 教程 by 落雨薄青衫
- 比赛现场视频：B 站 UP 主 翔林魔主

### 通信协议（简化版）

以下是简化版 RoboMaster 2026 客户端通信协议中的部分消息定义。

```proto
// 下行：服务器 -> 客户端；上行：客户端 -> 服务器

// 下行：比赛全局状态（发送频率 5Hz）
message GameStatus {
    // current_stage 枚举值：
    // 0:未开始比赛, 1:准备阶段, 2:十五秒裁判系统自检阶段,
    // 3:五秒倒计时, 4:比赛中, 5:比赛结算中
    uint32 current_round = 1;      // 当前局号（从 1 开始）
    uint32 total_rounds = 2;       // 总局数
    uint32 red_score = 3;          // 红方得分
    uint32 blue_score = 4;         // 蓝方得分
    uint32 current_stage = 5;      // 当前阶段
    int32 stage_countdown_sec = 6; // 当前阶段剩余时间
    int32 stage_elapsed_sec = 7;   // 当前阶段已过时间（秒）
    bool is_paused = 8;            // 是否暂停
}

// 下行：机器人实时数据（发送频率 10Hz）
message RobotDynamicStatus {
    uint32 current_health = 1;         // 当前血量
    float  current_heat = 2;           // 当前发射热量（太高会导致扣血）
    uint32 remaining_ammo = 3;         // 剩余允许发弹量
    uint32 current_buffer_energy = 4;  // 当前缓冲能量
    bool   is_out_of_combat = 5;       // 是否脱战
    uint32 current_experience = 6;     // 当前经验值
    uint32 experience_for_upgrade = 7; // 距离下一次升级仍需获得的经验
}

// 上行：传输鼠标输入（发送频率 75Hz）
message MouseControl {
    int32 mouse_x = 1;          // 鼠标 x 轴移动速度，负值标识向左移动
    int32 mouse_y = 2;          // 鼠标 y 轴移动速度，负值标识向下移动
    int32 mouse_z = 3;          // 鼠标滚轮移动速度，负值标识向后滚动
    bool left_button_down = 4;  // 左键是否按下（false=抬起, true=按下）
    bool right_button_down = 5; // 右键是否按下（false=抬起, true=按下）
    bool mid_button_down = 6;   // 中键是否按下（false=抬起, true=按下）
}

// 上行：地面机器人选择性能体系（发送频率 1Hz）
message RobotPerformanceSelectionCommand {
    uint32 shooter = 1; // 发射机构性能体系，枚举值：1:冷却优先, 2:爆发优先
    uint32 chassis = 2; // 底盘性能体系，枚举值：1:血量优先, 2:功率优先
}
```

### Part A - 界面实现

操作员在紧张的比赛中需要快速获取信息、做出判断。请你设计并实现叠加在图传画面上的界面元素。

1. 操作员需要看到当前的比赛状态和机器人关键数据。展示哪些信息、如何布局，由你根据通信协议自行设计。
2. 比赛节奏很快，操作员没有时间仔细看数字。关键数据的状态变化必须要能让操作员一眼辨别。
3. 操作员需要能在界面上向机器人下达控制指令。
4. 比赛数据会实时变化，你的界面需要能迅速地响应这些变化；操作员的控制操作也应能被外部获知。
5. 图传画面是操作员观察赛场、瞄准敌方的唯一途径。界面的排版与视觉展现必须最大程度地保留视野完整性，同时还要确保所有信息在赛场复杂、明暗交替的图传背景上依然清晰易读。
6. 状态变化以及与控件的交互，应配备合理的视觉动效，以辅助操作员在激烈对抗中做出本能反应。
7. 请自行在网上寻找一张第一视角的 RoboMaster 赛场图传截图作为交互界面的全局背景，你的所有界面元素都必须直接叠加在此图片上方进行开发与演示。

### Part B - 接入 C++ 后端

你的队友在 C++ 端已经处理好了与服务器的通信，拿到了各项比赛数据。现在需要你设计一个中间层，让你的界面能安全地用上这些数据，并且把操作员的操作传回给通信模块。

请让你的界面实际运行并演示以下效果：

1. 编写一个 C++ 类作为 Part A 界面的数据来源，并接入你的 QML 界面。
2. QML 界面运行在主线程（GUI 线程），而数据产生在后台子线程。请你在 C++ 端开启一个后台子线程模拟比赛数据变化（例如用定时器让血量逐渐减少），你的界面应能实时、安全且不卡顿地反映这些变化。
3. 当操作员在界面上与控件交互或移动点击鼠标时，C++ 端在控制台输出对应的指令信息。
4. 当操作员操作过快时，你将怎么处理这种情况？

补充说明：

1. 不需要实现真正的网络通信，不需要使用真实的 Protocol Buffer 生成实际消息体，用模拟数据即可。重点是让整条链路跑通。
2. 请使用 CMake 作为项目工程的构建系统，请不要使用 QMake。

## Q2 - 图传帧数据内存管理（附加题）

### 背景

在 Q1 中，你实现了操作员客户端界面。现在我们来关注客户端的另一个核心模块：图传。

机器人摄像头以 `1080p 60FPS` 回传 HEVC 编码视频流，每秒需要处理数 MB 的视频数据。如果每收到一帧就申请一块新内存、处理完就释放，频繁的内存申请和释放会带来严重的性能问题。

因此，我们需要一个帧缓冲池：程序启动时一次性准备好若干固定大小的内存块，供后续反复使用。请你阅读题目，尝试设计出符合下文需求的帧缓冲池。

本题主要探讨你的架构设计思维和对资源生命周期的理解。

1. 不要求写出绝对能通过编译与运行的完整实现。
2. 如果你写不出完整实现，只需要写出核心类的函数签名（API 接口），配合伪代码或文字描述内部逻辑即可。
3. 请优先尝试使用 C++ 语言完成。
4. 如果你熟悉 C++，强烈建议思考如何利用现代 C++ 特性（如 RAII、智能指针 `std::unique_ptr` / `std::shared_ptr`）来优雅地解决问题。

### Part A - 帧缓冲池

需求：

1. 程序启动时预先分配固定数量的内存块（例如 10 个），每块大小相同。
2. 使用者可以申请从池中借出一个可用内存块。如果池中没有空闲块，应当让申请者知道。
3. 内存块的归还机制由你设计。需要确保：已借出的内存块最终都能回到池中，不会出现内存块永久丢失。
4. 一块内存在同一时间只能有一个使用者。

### Part B - 模拟图传数据流

需求：

1. 接收端不断从帧缓冲池中借出内存块，模拟写入帧数据（可用 `sleep` 模拟耗时），然后将内存块传递给处理端。
2. 处理端接收内存块，模拟处理（同样可用 `sleep` 模拟），处理完后将内存块归还到池中。
3. 接收和处理需要并发/多线程执行（你需要考虑多个线程同时操作缓冲池时，如何保证线程安全）。
4. 程序退出时，所有工作都需要正确停止，所有资源都能被正确清理。
